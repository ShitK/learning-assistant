# AGENTS.md

## 1. 编码前先思考

**不要假设。不要掩盖困惑。明确说出取舍。**

实现之前：
- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解，列出来，不要默默选择其中一种。
- 如果有更简单的做法，要说出来。必要时要提出反对意见。
- 如果某件事不清楚，就停下来。说清楚哪里让你困惑，然后提问。

## 2. 简单优先

**用能解决问题的最少代码。不要写推测性的东西。**

- 不添加需求之外的功能。
- 不为只使用一次的代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行，而本来 50 行就能解决，就重写并简化。

本项目例外：外部模型、图片上传、JSON Schema、localStorage 和浏览器状态恢复相关失败路径不是“过度防御”，而是演示稳定性的基本要求。Kimi 超时、未配置 API Key、非法 JSON、图片过大、样例数据缺失等路径必须有清晰处理。

问问自己：“资深工程师会不会觉得这太复杂？” 如果答案是会，就简化。

## 3. 精准修改

**只改必须改的地方。只清理自己造成的问题。**

编辑现有代码时：
- 不要顺手“改进”相邻代码、注释或格式。
- 不要重构没有坏掉的东西。
- 遵循现有风格，即使你个人会用不同写法。
- 如果发现无关的废弃代码，提出来，不要直接删除。

当你的修改产生了孤立代码时：
- 移除由你的修改导致不再使用的 import、变量、函数。
- 除非被要求，否则不要移除原本就存在的废弃代码。

检验标准：每一行改动都应该能直接追溯到用户的请求。

补充约定：

```text
一个任务 → 一个分支
一个分支 → 一个主题
一个提交 → 一个可解释的变化
```

不要在同一次改动中混入无关重构、样式调整、格式化或代码清理。发现无关问题时先说明或记录，除非用户明确要求，否则不直接修改。

## 4. 目标驱动执行

**定义成功标准。循环推进直到验证通过。**

把任务转化成可验证的目标：
- “添加校验” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写一个能复现问题的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，说明一个简短计划：

```text
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰的成功标准能让你独立循环推进。模糊的标准（比如“让它能用”）会导致不断需要澄清。

## 5. 语言约定

- 项目文档、需求说明、设计说明、任务计划、审查意见默认使用中文。
- 代码注释在必要时使用中文，重点解释业务意图、复杂逻辑或容易误解的取舍。
- 不要为显而易见的代码添加注释。
- 代码标识符、文件名、接口字段名、第三方 API 名称保持项目既有风格；不要为了中文化而改动它们。
- 如果引用外部英文错误信息、库名、命令、配置项或协议字段，保留原文。

## 6. AI 协作流程

对于较复杂、有风险或涉及代码实现的任务，默认采用以下协作节奏：

```text
需求 -> Git 分支 -> Codex 规划 -> Codex 实现 -> Codex 自测
   -> Claude Code 审查 -> Codex 修复 -> 再测 -> 合并 main
```

执行约定：
- 较复杂或有风险的任务应先创建 Git 分支，不直接在 main 上修改。
- Codex 负责阅读上下文、明确假设、制定计划、实现代码、运行相关自测，并说明验证结果。
- Claude Code 主要负责代码审查，重点关注 bug、回归风险、测试缺口、安全或数据丢失风险、无关改动。
- Claude Code 审查后的建议写入 `docs/reviews/YYYY-MM-DD-任务名-review.md`。
- Codex 根据审查意见逐项修复，修复后重新运行相关测试，并说明哪些问题已处理、哪些保留不处理以及原因。
- Codex 在“实现完成 + 自测通过 + Claude Code 审查修复完成 + 再测通过”后，默认创建一个本地 Git commit 作为交付检查点；除非用户明确要求不提交。
- 创建本地 commit 前必须先展示并核对 `git status --short` 和提交范围，只精确 stage 本任务相关文件；不得使用 `git add .` 这类容易混入无关文件的命令。
- 本地 commit 不得包含 `.env*`、API Key、用户明确要求忽略的 review 文档、未确认的用户改动或与本任务无关的文件。
- 如果工作区存在用户改动或无关未跟踪文件，Codex 应保持它们不被 stage；若无法判断文件归属，应先询问再提交。
- `docs/reviews/*.md` 默认只作为本地审查材料保留，不 stage、不 commit、不 push；只有用户明确要求提交某一份审查报告时，才纳入 Git 历史。
- PR 或最终说明中只总结“已进行本地 Claude Code 审查、已修复/保留哪些问题”，不上传完整审查报告到 GitHub。
- 如果用户没有明确要求创建 PR，复杂或高风险任务完成本地 commit 和本地审查后，默认采用“feature 分支本地合并到 `main`，在 `main` 上重新自测，再 push `main`”的集成流程。
- 采用本地合并流程时，Codex 必须先确认 feature 分支已提交、`main` 已更新到最新、合并后 `npm test` / `npm run lint` / `npm run build` 等相关验证通过，再推送 `main`。
- 只有当用户明确要求 PR、需要远程协作审查，或任务风险高到需要保留 GitHub 审查记录时，才默认走“push feature branch -> create PR”的流程。
- 简单文档修改、小范围无风险改动可以简化流程，但仍应保持改动范围清晰。

## 7. 本项目特殊约束

本项目是 **错因地图 MathTrace**，一个用于黑客松的高中数学错题诊断 Agent。当前 P0 是稳定 Web Demo，不是完整商业系统。

### 7.1 技术栈和演示路径

- 技术栈默认采用 Next.js App Router + TypeScript + Tailwind CSS。
- 数学公式渲染使用 KaTeX。
- API 入参、模型响应、mock JSON 的运行时校验优先使用 Zod；如果暂未引入 Zod，必须至少保留明确的 TypeScript 类型和边界校验函数。
- P0 正式演示路径固定为 `sample_diagnosis`，使用预标注样例题和 mock 学生画像。
- `image_diagnosis`、Kimi 图片识别、LLM 动态生成变式题、`/api/confirm` 都属于 P1，不要提前塞进 P0。
- P0 只保留一个核心接口：`POST /api/diagnose`。
- 不做 SSE 流式输出，不做多接口串行 Agent，不做复杂多智能体通信。

### 7.2 Next.js 和 RSC 约束

- 默认使用 Server Components；只有需要浏览器 API、React state、事件处理、localStorage 或动画状态时才添加 `'use client'`。
- Client Component 尽量放在组件树叶子节点，避免把整页都变成客户端渲染。
- 禁止在 Client Component 中直接读取服务端环境变量、调用 Kimi、访问文件系统或执行服务端数据读取。
- API Route 放在 `app/api/**/route.ts`，显式处理 HTTP 方法、入参校验、错误响应和状态码。
- Kimi API 调用只能发生在服务端 API Route 或服务端工具函数中。

### 7.3 TypeScript 和代码结构

- 使用准确示例编写简洁、技术化的 TypeScript；示例代码应能直接表达真实数据结构和边界，不写空泛伪代码。
- 全部业务代码使用 TypeScript，避免 `any`；未知外部输入使用 `unknown` 后再通过 Zod 或类型守卫收窄。
- 导出的函数、API handler、核心数据转换函数应显式标注返回类型。
- 对象数据结构优先使用 `interface` 或清晰的 `type`；避免为业务枚举使用 TypeScript `enum`，优先使用 `as const` 对象或字面量联合类型。
- 组件优先使用具名 `function` 定义，并使用 named exports，避免默认导出造成重命名混乱。
- 变量命名使用有辅助动词的描述性命名，例如 `isDiagnosing`、`hasFallback`、`canStartDiagnosis`。
- 使用函数式和声明式编程模式，避免使用类；复用逻辑优先抽成小函数、数据映射或模块化工具函数。
- 优先通过迭代、组合和模块化减少重复代码，不复制粘贴相似分支。
- 文件结构规范：imports 在最前；导出的主组件或主函数尽量靠近文件开头；本文件私有 helper、常量和子组件放在其后，除非依赖顺序要求相反。

### 7.4 数据和状态

- 内部数据 key 使用 snake_case `id`，例如 `parameter_classification`。
- 前端展示中文名时通过知识库中的 `display_name` 映射，不要把中文名当作内部 key。
- P0 状态持久化使用前端 localStorage；后端 API Route 保持无状态。
- `student_profile`、`mistake_history`、`memory_delta` 必须与 PRD 中的 Schema 保持一致。
- 修改知识点、错因标签、样例题、mock 画像、Zod schema、TypeScript 类型或 API 响应结构时，必须同步检查 PRD。

### 7.5 安全和隐私

- Kimi API Key 只能放在服务端环境变量中，不得出现在前端代码、mock 数据、日志或提交历史里。
- P0 不采集真实学生身份信息，默认使用 `demo_student_001`。
- 上传图片和错题内容可能包含未成年人学习数据，调试日志中不要输出完整图片 base64 或敏感原文。

### 7.6 范围控制

- 不新增老师端、家长端、多科目、RAG、真实数据库、登录注册、支付、部署流水线等非 P0 功能，除非用户明确要求。
- 不默认引入 Shadcn UI、Radix UI 或其他组件库；只有当它明显减少实现成本且不拖慢交付时再引入。
- 发现可扩展想法时记录为 roadmap，不要直接实现。
- 页面优先服务 3-5 分钟演示：样例题诊断、Agent 步骤、错因报告、画像变化、变式题、7 天建议。

### 7.7 架构先行与模块边界

涉及目录结构、模块边界、数据契约、模型 provider、数据库、学生画像、localStorage、确认链路或持久化策略的任务，默认先写 plan，不直接实现。

计划中必须说明：

- 目标是什么。
- 明确不做什么。
- 涉及哪些文件和模块。
- 数据如何流动。
- 哪些已有行为不能被破坏。
- 如何验证没有行为回归。

架构清理必须保持小步、单主题、可验证。不要为了“更工程化”一次性做大范围目录重组、抽象升级或风格调整。

当前项目模块边界：

- `app/api/**`：只作为 HTTP 入口，负责请求解析、状态码、错误响应和调用 service；不要写复杂业务规则。
- `components/**`：负责展示、交互和状态连接；不要直接访问数据库、模型 provider、文件系统或 service role key。
- `src/lib/shared/**`：只能放 browser-safe、无副作用、跨模块复用的类型、常量、类型守卫和纯函数；禁止依赖 provider、persistence、server-only 环境变量或 Node-only API。
- `src/lib/providers/**`：只封装外部模型协议差异；不能决定 `memory_delta`、`student_profile` 或持久化策略。
- `src/lib/vision-extraction/**`：只负责图片抽取契约、parser、debug summary 和 provider 输入输出边界；视觉模型不得输出画像、错因频次、`memory_delta` 或最终标准解法。
- `src/lib/diagnosis/**`：负责确定性诊断 pipeline、证据策略、画像增量和确认流程编排。
- `src/lib/persistence/**`：只封装数据库读写、RPC payload 和服务端持久化边界；不要承载错因诊断规则。
- `src/lib/demo/**`：只负责 demo 状态恢复、localStorage 包装和本地演示状态，不代表云端完整学生画像。

当前项目架构红线：

- `sample_diagnosis` 是稳定演示路径，不能被数据库、真实模型 provider 或架构重构破坏。
- 前端不能直连 Supabase，也不能读取 service role key。
- 模型输出一律视为不可信输入，必须经过 parser、字段白名单、类型守卫或业务规则收口。
- 文本分析模型只能增强展示文本，不能决定画像写入。
- `StudentProfile` 运行时校验以 shared guard 为准，不要再复制多套画像校验逻辑。
- localStorage 目前只是 demo 画像恢复来源，不等于完整云端学生画像。

## 8. 文档维护约束

项目采用轻量文档策略：文档服务于开发决策、协作理解和后续维护，不为形式完整而创建空文档或无实际内容的文档。

### 8.1 开发前

复杂或有风险的任务在实现前必须说明：

- 目标是什么
- 当前假设是什么
- 改动边界是什么
- 验收方式是什么
- 哪些已有行为不能被破坏

实现前优先对齐 PRD：`docs/superpowers/specs/2026-05-28-math-mistake-agent-prd.md`。

如果任务会改变 API 契约、数据模型、Agent 流程、mock 数据、Kimi 调用方式、localStorage 状态模型或核心模块边界，应同步更新 PRD；当实现明显超出 PRD 或需要更稳定的长期说明时，再新增 `docs/ARCHITECTURE.md`。

### 8.2 开发中

开发过程中只同步更新与本次改动直接相关的文档，不做无关整理。

出现以下变化时，必须同步更新对应文档：

- 业务规则变化：优先更新 PRD；必要时新增 `docs/DOMAIN.md`
- 模块边界或核心数据流变化：优先更新 PRD；必要时新增 `docs/ARCHITECTURE.md`
- 启动方式、环境变量、部署方式变化：更新 `README.md` 或部署相关文档
- 接口字段、知识库 JSON、mock 数据、localStorage schema 或权限规则发生变化：更新 PRD 或当前项目中对应的说明文档

如果实现过程中发现原计划不准确，应更新任务计划或在最终说明中明确偏差与原因。

### 8.3 ADR 使用规则

只有重要技术取舍才需要写 ADR，路径使用 `docs/adr/`。

满足以下任一条件时，应写 ADR：

- 该决策会长期影响架构
- 存在多个可行方案且取舍明显
- 未来维护者可能会追问“为什么这样做”
- 决策错误会导致较高迁移、数据、安全或回归成本

普通样式调整、小范围页面改动、一次性工具函数、无争议实现细节不写 ADR。

### 8.4 合并前文档收口

复杂或高风险任务在合并前必须检查文档是否与最终代码一致。

至少检查：

- `README.md` 是否需要更新启动、配置、环境变量或命令
- PRD 是否覆盖最终 API 契约、数据模型、演示路径和 P0/P1 边界
- `docs/DOMAIN.md` 是否需要覆盖最终业务规则
- `docs/ARCHITECTURE.md` 是否需要反映最终模块边界和数据流
- 是否产生了需要 ADR 记录的重要取舍

如果本次改动不影响文档，应在最终说明中明确写出：`本次改动不涉及文档更新`。

### 8.5 文档强度分级

根据任务风险决定文档要求：

- 小改动：说明目标和验证结果即可，不强制新增文档
- 中等功能：需要任务计划，并同步更新 PRD 或相关 `DOMAIN` / `ARCHITECTURE` 内容
- 高风险功能：必须开分支、写计划、自测、审查，并在合并前完成文档收口

高风险范围包括但不限于：

- Kimi API 调用、prompt 和 JSON Schema
- API Key、环境变量和日志安全
- 学生画像、错题历史和 localStorage 状态
- `memory_delta` 和掌握度计算
- 知识库 JSON、错因标签和样例题数据
- 图片上传、压缩和错误处理
- 数据删除、重置或迁移
- 数据库结构（如果后续引入）

### 8.6 面试叙事文档维护

本项目同时服务于求职展示。除了 PRD、Roadmap、计划和审查文档，还应持续维护面试叙事文档：

```text
interview/mathtrace-project-narrative.md
```

维护目标：

- 把每个重要阶段沉淀成“可面试表达”的项目叙事。
- 记录功能价值、关键设计、技术取舍、可追问问题和推荐回答。
- 如果某项技术决策带来性能、响应速度、稳定性、资源占用或构建/测试效率收益，必须记录“性能收益”和可验证依据。
- 如果后续功能扩展改变了早期性能收益或技术取舍，应回到对应旧阶段更新说明，标注原收益是否仍成立、被替代或转化为新的取舍。
- 用真实文件、测试、文档、验证命令或 smoke test 建立证据链。
- 帮助后续复盘：这个项目为什么这样做，而不是只记录做了什么。

当完成以下类型的阶段时，应检查并更新该文档：

- 新增或完成一个用户可感知的功能阶段。
- 改变 API 契约、数据模型、Agent Pipeline、模型 provider、图片上传、localStorage 或学生画像边界。
- 做出重要技术取舍，例如不引入某个框架、选择确定性 pipeline、限制模型输出边界，或通过压缩、缓存、减少网络调用等方式改善性能。
- 完成一次高风险功能的 Claude Code 审查、修复和最终验证。
- 完成一次能影响面试叙事的真实验证，例如真实模型 smoke test、视觉验证或稳定性回归。

小范围错别字、无行为变化的样式微调、临时调试或纯内部清理，不强制更新面试叙事文档。

新增阶段时默认使用以下结构：

```md
## N. 阶段名称

### 当前状态
已完成 / 已合并 / 已验证 / 有已知缺口，可组合描述。

### 功能价值
这个阶段解决了什么真实问题？为什么对用户、演示或业务重要？

### 关键设计
核心模块怎么拆？数据怎么流？边界在哪里？

### 技术决策与取舍
为什么选这个方案？没选哪些方案？代价是什么？

### 性能收益（如适用）
这个阶段是否改善了响应速度、资源占用、稳定性、构建/测试效率或模型调用成本？有没有数据、命令、截图、smoke test 或代码证据？

### 面试官可能怎么问
列 5-8 个真实问题。

### 推荐回答
用第一人称回答，尽量像项目作者在面试里自然表达。

### 可能被继续追问
面试官可能沿着哪些点继续深挖。
```
