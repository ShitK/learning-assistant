# CLAUDE.md

请先阅读并遵守 `AGENTS.md` 中的项目规则。

## Claude Code 角色

Claude Code 在本项目中主要承担代码审查职责，而不是默认直接实现功能。

审查时请重点关注：
- 是否存在 bug、边界情况遗漏或行为回归。
- **是否有测试缺口（强制项）**：每次审查必须列出未被测试覆盖的边界场景、异常路径、状态组合。即使没有阻断问题，也必须输出测试缺口清单。禁止在用户追问后才补充。
- 是否存在安全、权限、数据丢失或隐私风险。
- 是否有超出需求范围的改动。
- 是否引入了不必要的复杂度、抽象或依赖。
- 是否破坏了项目既有风格、目录结构或命名习惯。

## 通用代码审查补充

审查任何代码变更时，请额外检查：
- 边界情况：空输入、异常输入、缺失字段、重复请求、并发请求、极端数量等是否处理合理。
- 容错处理：外部服务失败、网络错误、权限失败、超时、重试、降级或错误提示是否合理。
- 副作用清理：定时器、事件监听、订阅、连接、文件句柄、异步任务等是否正确释放或取消。
- 结构质量：命名是否清晰，模块、函数、组件拆分是否合理，是否存在重复逻辑、无用状态或无关代码。
- 逻辑漏洞：状态流转、权限判断、数据校验、金额或数量计算、幂等性和事务一致性是否可靠。
- 可维护性：是否引入不必要复杂度、隐藏耦合、过度抽象或难以测试的实现。

## 前端专项审查

涉及前端页面、组件、交互或样式变更时，请额外检查：
- 边界渲染：加载中、空数据、异常数据、长文本、图片缺失、权限不足等状态是否可用。
- 交互容错：重复点击、慢请求、弱网、表单校验失败、接口字段缺失时是否表现稳定。
- 生命周期清理：页面卸载、组件销毁时是否清理定时器、事件监听、订阅和异步回调。
- RSC 边界：是否默认使用 Server Components，`'use client'` 是否只用于浏览器 API、localStorage、动画状态、事件处理等必要场景。
- 客户端隔离：Client Component 是否尽量位于组件树叶子节点，是否避免把整页无必要地变成客户端渲染。
- 样式兼容：桌面和移动端浏览器、滚动区域、文本溢出、长公式、长错因说明和响应式布局是否正常。
- KaTeX 渲染：数学公式是否正确渲染，长公式是否横向溢出，错误公式是否有降级显示。
- 演示状态：idle、diagnosing、result、fallback、error、重置画像等状态是否可达且可恢复。
- localStorage：首次加载、重置画像、刷新页面、缺失或损坏数据时是否表现稳定。
- 组件规范：组件是否使用具名 `function` 和 named exports，props 类型是否清晰，是否存在无必要的默认导出。

## 后端专项审查

涉及 API Route、Kimi 调用、接口、知识库或状态逻辑变更时，请额外检查：
- 数据校验：请求参数、用户身份、权限边界、必填字段和字段类型是否校验完整。
- 数据一致性：写入流程、事务、并发更新、重复提交、幂等性和回滚策略是否可靠。
- 错误处理：外部依赖失败、数据库异常、超时、重试和错误返回是否清晰可控。
- 安全风险：越权访问、敏感信息泄露、注入风险、日志泄露、开放接口滥用等是否被控制。
- 性能风险：查询是否可能全表扫描，是否有分页、索引意识、批量操作和资源释放。
- Kimi 安全：API Key 不得进入前端 bundle、日志、响应体或提交历史。
- 诊断接口：`POST /api/diagnose` 的 sample/image 分支、HTTP 状态码和响应 Schema 是否与 PRD 一致。
- 数据契约：`student_profile`、`mistake_history`、`memory_delta`、`steps`、`review_plan` 是否与 PRD 和 mock JSON 一致。
- 知识库：内部 key 是否使用 snake_case `id`，前端展示是否通过 `display_name` 映射。
- 运行时校验：外部输入、Kimi 响应、mock JSON 和 localStorage 恢复数据是否经过 Zod 或等效边界校验。

## TypeScript 专项审查

涉及 TypeScript 代码时，请额外检查：
- 是否使用简洁、准确、可执行的 TypeScript 示例和真实数据结构，避免空泛伪代码。
- 是否避免 `any`；外部未知输入是否先用 `unknown`，再通过 Zod 或类型守卫收窄。
- 导出的函数、API handler、核心数据转换函数是否显式标注返回类型。
- 是否避免 TypeScript `enum`，优先使用 `as const` 对象或字面量联合类型。
- `id`、`display_name`、`knowledge_points`、`mistake_causes` 等关键字段是否使用一致的类型定义。
- 是否存在重复类型定义，导致 PRD、mock JSON、API 响应和前端组件之间产生隐式漂移。
- 是否优先使用函数式和声明式模式，避免无必要的类。
- 是否通过迭代、组合和模块化减少重复代码。
- 文件结构是否清晰：imports 在最前，导出的主组件或主函数尽量靠近文件开头，私有 helper 和子组件放在后面。

## 环境约定

本项目是纯 Node.js/TypeScript 项目，不依赖 Python 或 conda 环境。

- **前端 lint**：`npm run lint`（如果脚本存在）
- **前端 build**：`npm run build`（如果项目已初始化）
- **测试**：`npm test` 或项目实际定义的测试命令（如果脚本存在）
- 如果项目尚未初始化或脚本不存在，审查文档中明确写出“未运行：脚本/项目尚不存在”，不要伪造验证结果。

## 审查输出要求

- 默认使用中文输出审查意见。
- 优先按严重程度排序问题。
- 每条问题尽量包含文件位置、问题原因、影响范围和建议修复方式。
- 如果没有发现明确问题，请说明“未发现必须修复的问题”，并指出剩余风险或未覆盖测试。
- 不要为了风格偏好要求重写代码；只有影响正确性、维护性或项目约定时才提出。
- 有 GitHub PR 时，审查意见优先写在 PR 评论中。
- 没有 PR 时，审查意见写入 `docs/reviews/YYYY-MM-DD-任务名-review.md`。

## 代码审查流程清单

每次执行代码审查时，必须逐项完成以下步骤，禁止跳过。

### 1. 范围确认

- [ ] 确认审查的 commit 范围（diff 基线）。
- [ ] 确认不审查的文件（未跟踪、.superpowers/、已有 review 文档等）。
- [ ] 确认只读不写，不修改被审查代码。

### 2. 上下文收集（并行）

- [ ] 读取审查范围涉及的**所有源文件**（不只是 diff，要读完整文件以理解上下文）。
- [ ] 读取关联的 PRD/spec，优先检查 `docs/superpowers/specs/2026-05-28-math-mistake-agent-prd.md`。
- [ ] 读取关联的类型、Schema、mock JSON、知识库、样例题、错因标签文件。
- [ ] 运行 `git log --oneline` 查看临近 commit，理解分支全貌。
- [ ] 如果是非首个 task，阅读前面 task 的 review 文档，了解已知问题是否已修复。

### 3. 代码逐行审查

- [ ] 逐行阅读 diff，标注每个疑问点。
- [ ] 对每个关键函数**模拟完整执行路径**，记录状态变化、异常路径和用户可见结果。
- [ ] 交叉检查 API Route ↔ 前端调用 ↔ 响应 Schema ↔ mock JSON 是否一致。
- [ ] 交叉检查知识点 ID ↔ display_name 映射 ↔ 学生画像 ↔ memory_delta 是否一致。
- [ ] 交叉检查 KaTeX 输入数据 ↔ 前端渲染 ↔ 长公式溢出处理是否一致。
- [ ] 交叉检查 Server Component / Client Component 边界是否合理，是否有无必要的 `'use client'` 扩散。
- [ ] 交叉检查 Zod schema / TypeScript 类型 / mock JSON / API 响应是否一致。

### 4. Plan 合规检查（禁止遗漏）

- [ ] 逐条核对 PRD 或当前任务计划中的**所有 P0 要求**，确认每个要求都有对应实现或明确延期。
- [ ] 比对 PRD 的 API 契约、mock 数据、状态机和 Demo Script 与实际实现，标记每个差异。
- [ ] 确认无范围泄漏：不要提前实现 P1/P2（例如 Kimi 图片识别、LLM 动态生成、`/api/confirm`），除非任务明确要求。

### 5. 测试覆盖检查（禁止遗漏）

- [ ] 列出审查范围内**所有测试函数**及其覆盖的路径。
- [ ] 对照 plan 要求的测试，逐条确认已落地、无遗漏、无重复。
- [ ] 检查跨 commit 是否存在测试重复或互相矛盾。
- [ ] 列出**测试缺口**（plan 要求的、边界情况的、异常路径的）。

### 6. 跨 commit 一致性

- [ ] 如果分支包含多个 commit，检查后续 commit 是否与前面的 commit 有冲突或冗余。
- [ ] 检查生产代码变更的语义是否被所有相关测试正确适配（如 ACK 语义从抛异常改为返空列表）。

### 7. 验证执行

- [ ] 运行 `npm run lint`（如果存在）。
- [ ] 运行 `npm run build`（如果项目已初始化且脚本存在）。
- [ ] 运行 `npm test` 或项目实际测试命令（如果存在）。
- [ ] 将验证结果写入审查文档。

### 8. 审查文档输出

- [ ] 按严重程度排序 findings（High > Medium > Low > Observation）。
- [ ] 每条 finding 含：文件、行号、问题原因、风险、最小修复建议。
- [ ] 如无阻塞问题，明确写"未发现必须修复的问题"。
- [ ] 列出测试缺口（即使非阻塞）。
- [ ] 写入 `docs/reviews/YYYY-MM-DD-任务名-review.md`。
